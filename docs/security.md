# Security

## Network Isolation

All three EC2 instances are placed in a private subnet with no public IP addresses. No internet gateway route exists for inbound traffic. The sole management channel is AWS SSM Session Manager, which uses outbound HTTPS (443) to SSM VPC endpoints — no inbound SSH port is opened on any security group.

Every security group allows inbound traffic only from `var.internal_cidr`. This single CIDR covers workstations, Fargate containers, and inter-VM traffic. Egress is restricted to port 443 (SSM endpoints and the Elastic APT repository).

| Security Group | Allowed Inbound Ports | Source |
|---|---|---|
| elasticsearch-sg | 9200/tcp | `var.internal_cidr` |
| kibana-sg | 5601/tcp, 8220/tcp | `var.internal_cidr` |
| apm-sg | 8200/tcp | `var.internal_cidr` |
| All | 443/tcp outbound | `0.0.0.0/0` |

No port 22 (SSH) is opened. No bastion host is required.

## TLS Design

All service-to-service and client-to-service communication is encrypted. Certificates are issued by the account's existing AWS Private Certificate Authority.

### Certificate Lifecycle (Terraform)

For each VM role (`elasticsearch`, `kibana`, `apm`):

1. A 4096-bit RSA private key is generated by the Terraform `tls` provider.
2. A Certificate Signing Request is built with the role's FQDN as CN and SAN. The Kibana cert includes both `kibana.<domain>` and `fleet.<domain>` as SANs since Fleet Server shares the same VM.
3. The CSR is signed by the Private CA via `aws_acmpca_certificate` (validity: 825 days).
4. The private key is stored in SSM Parameter Store as a `SecureString` (KMS-encrypted) at `/elastic/<role>/tls/key`.
5. The signed certificate is stored at `/elastic/<role>/tls/cert`.
6. The CA certificate chain is stored once at `/elastic/common/tls/ca_chain` and shared across all roles.

### Certificate Delivery (Ansible)

The `common` Ansible role runs on every VM before any service-specific role. It:

1. Reads the key, cert, and CA chain from SSM Parameter Store using the AWS CLI (the instance IAM role grants `ssm:GetParameter` on `/elastic/*`).
2. Writes them to `/etc/elastic/tls/` with `0640` permissions, owned by `root` and group-readable by the service user (`elasticsearch`, `kibana`, or `elastic-agent`).

### Where TLS Is Applied

| Service | Protocol | Config |
|---|---|---|
| Elasticsearch HTTP | HTTPS on 9200 | `xpack.security.http.ssl.*` |
| Elasticsearch transport | TLS on 9300 | `xpack.security.transport.ssl.*` |
| Kibana | HTTPS on 5601 | `server.ssl.*` |
| Fleet Server | HTTPS on 8220 | passed as CLI flags to `elastic-agent install` |
| APM Server (via Agent) | HTTPS on 8200 | configured in Fleet APM integration policy |

## IAM and Least Privilege

Each VM has its own IAM role (not a shared role). Each role has:

- `AmazonSSMManagedInstanceCore` — the AWS managed policy required for SSM Session Manager.
- An inline policy granting `ssm:GetParameter`, `ssm:GetParameters`, and `ssm:GetParametersByPath` scoped to `arn:aws:ssm:<region>:*:parameter/elastic/*` only.

No broader AWS permissions are granted. EC2 instance metadata access is not restricted further, but no role has EC2 API, S3, or other service permissions.

## Credential Management

Elastic Stack credentials are never stored in configuration files or version control.

| Credential | Storage | Path |
|---|---|---|
| `elastic` superuser password | SSM SecureString | `/elastic/credentials/elastic_password` |
| `kibana_system` password | SSM SecureString | `/elastic/credentials/kibana_system_password` |
| Fleet Server service token | SSM SecureString | `/elastic/credentials/fleet_server_token` |
| APM enrollment token | SSM SecureString | `/elastic/fleet/apm_enrollment_token` |
| Kibana keystore | Local keystore on VM | `/etc/kibana/kibana.keystore` |

The Elasticsearch role resets `elastic` and `kibana_system` passwords on first provisioning and pushes them to SSM. Kibana reads `kibana_system` from SSM during provisioning and injects it into the Kibana keystore — it does not appear in `kibana.yml`.

## Terraform State

The Terraform `tls_private_key` resource stores private keys in the state file in plaintext. This is a known limitation of generating TLS material in Terraform.

**Mitigation required before use in a shared or production environment:**

- Configure a remote S3 backend with server-side encryption using a KMS customer-managed key.
- Apply a restrictive S3 bucket policy: deny `s3:GetObject` to any principal that is not the CI/CD role or the deploying user.
- Enable S3 versioning and MFA delete on the state bucket.
- Consider migrating certificate issuance to a separate, tightly-controlled Terraform workspace or to an external secrets manager workflow if the threat model demands it.

## What Is Not in Version Control

The `.gitignore` blocks:

- `*.tfvars` — contains VPC IDs, CIDRs, domain names, ARNs
- `terraform.tfstate`, `*.tfstate.*` — contains private keys and credentials
- `ansible/inventory/hosts.yml` — generated file containing instance IDs and IPs
- `*.pem`, `*.key` — TLS material should never be checked in
