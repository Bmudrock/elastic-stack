# Architecture

## Purpose

This project deploys an Elastic Stack observability platform on AWS to collect and visualise logs, metrics, and APM traces from two application scenarios, each running 37 Fargate containers (74 agents total).

## Components

### Elasticsearch (`r6i.2xlarge`)

Single-node cluster. Receives all logs, metrics, and APM traces indexed by Kibana-managed agents and the APM Server agent. Stores data on a dedicated 1 TB gp3 EBS volume mounted at `/var/lib/elasticsearch` (XFS).

X-Pack security is enabled. All HTTP and transport traffic uses TLS. A zero-replica index template is applied on first boot to prevent unassigned shards on a single-node cluster.

### Kibana + Fleet Server (`m6i.xlarge`)

Two services on one VM. Kibana provides the UI and API layer. Fleet Server runs as a managed `elastic-agent` process on the same host and acts as the central control plane for all enrolled agents (Fargate sidecars + the APM VM agent).

Fleet is co-located with Kibana at this scale (74 agents) to avoid the cost and operational overhead of a fourth VM.

### APM Server (`m6i.large`)

Runs `elastic-agent` enrolled in a Fleet APM integration policy. This is the modern Elastic 8.x approach — APM Server functionality is delivered through Elastic Agent rather than the legacy standalone `apm-server` binary. The agent is enrolled during Ansible provisioning using an enrollment token generated by Fleet and stored in SSM Parameter Store.

## Data Flow

```
Fargate task
  └─ Elastic Agent sidecar
       ├─ logs/metrics ──► Fleet (8220) ──► Elasticsearch (9200)
       └─ APM traces   ──► APM Server (8200) ──► Elasticsearch (9200)

Workstations
  └─ browser ──► Kibana (5601) ──► Elasticsearch (9200)
```

## Sizing Basis

| Input | Value |
|---|---|
| Container count | 74 (37 per scenario) |
| Container profile | 1 vCPU, 2 GB RAM (medium) |
| Telemetry | Logs + metrics + APM traces |
| Retention | 7 days |

See [terraform.md](terraform.md) for per-VM sizing rationale.

## Key Constraints

- **Single-node Elasticsearch** — no high availability. A node failure means the stack is unavailable until the instance recovers. Acceptable for an internal observability platform; move to a 3-node cluster if uptime SLAs tighten.
- **No public IPs** — all three VMs are in a private subnet. Management is exclusively via AWS SSM Session Manager.
- **Inbound traffic locked to one CIDR** — `var.internal_cidr` is the sole trusted source for all security group rules. Fargate containers and workstations must be reachable from within that range.
- **7-day hot retention only** — no warm/cold tiers. Older data is deleted by ILM policy. Extend `elasticsearch_ebs_size_gb` and adjust ILM if longer retention is required.
- **Ubuntu 26.04 APT support** — Elastic's APT repository uses the `stable` distribution label (not the Ubuntu codename), so this is not expected to be an issue. The `ami_id` variable makes it straightforward to fall back to Ubuntu 24.04 LTS if any package-level incompatibilities arise.
