# Ansible

## Overview

The `ansible/` directory configures the three EC2 instances provisioned by Terraform. It installs Elastic Stack services, mounts and formats EBS volumes, deploys TLS certificates, bootstraps credentials, sets up Fleet Server, and enrolls the APM agent.

Ansible connects to all hosts exclusively via the **`community.aws.aws_ssm` connection plugin** — no SSH, no bastion. The instance IDs in the generated inventory are the SSM targets.

## Prerequisites

```bash
# Install required Ansible collections
ansible-galaxy collection install -r ansible/requirements.yml

# AWS credentials must be active with permissions for:
#   ssm:StartSession, ssm:GetParameter, ssm:PutParameter
#   ec2:DescribeInstances (used by SSM connection plugin)
```

The `session-manager-plugin` binary must be installed on the Ansible controller. See [AWS documentation](https://docs.aws.amazon.com/systems-manager/latest/userguide/session-manager-working-with-install-plugin.html).

## File Structure

```
ansible/
├── requirements.yml          # amazon.aws and community.aws collections
├── ansible.cfg               # inventory path, SSM-friendly defaults
├── inventory/
│   └── hosts.yml             # generated by terraform apply (git-ignored)
├── group_vars/
│   ├── all.yml               # shared vars: TLS path, SSM prefix, Fleet port
│   ├── elasticsearch.yml     # heap size, ports, data dir, cluster name
│   ├── kibana.yml            # Kibana port, data dir
│   └── apm_server.yml        # APM port
├── playbooks/
│   ├── site.yml              # imports all three playbooks in order
│   ├── elasticsearch.yml
│   ├── kibana.yml
│   └── apm_server.yml
└── roles/
    ├── common/
    ├── elasticsearch/
    ├── kibana/
    ├── fleet_server/
    └── apm_server/
```

## Roles

### `common`

Runs on every host before any service role. Responsibilities:

- Installs prerequisites (`apt-transport-https`, `curl`, `gnupg`, `xfsprogs`, `awscli`)
- Adds the Elastic APT repository (signed key, `stable` distribution — not Ubuntu-codename-specific)
- Creates `/etc/elastic/tls/` with `0750` permissions
- Fetches the TLS private key, certificate, and CA chain from SSM Parameter Store
- Writes cert material to `/etc/elastic/tls/` with appropriate ownership for the service user
- Verifies that the SSM agent is running

The `stable` distribution label in the Elastic APT repository means Ubuntu 26.04's codename does not need to be supported explicitly by Elastic — this sidesteps any new-release APT repo lag.

### `elasticsearch`

- Installs `elasticsearch` from the Elastic APT repo
- Adjusts TLS directory permissions for the `elasticsearch` group
- Formats and mounts `/dev/xvdf` as XFS at `/var/lib/elasticsearch` (skipped if already formatted — idempotent)
- Renders `elasticsearch.yml` and `jvm.options.d/heap.options` from templates
- Enables and starts `elasticsearch` via systemd
- Waits for the HTTP API to respond (accepts 200 or 401 — either means ES is up)
- Resets the `elastic` and `kibana_system` built-in user passwords and stores both in SSM Parameter Store as SecureStrings

**Heap configuration**: set to `32g` (`-Xms32g -Xmx32g`) in `group_vars/elasticsearch.yml`. 32 GB is the practical upper limit before the JVM exits compressed-oops mode, which would negate the memory savings.

**Zero-replica template**: after first boot an index template is applied that sets `number_of_replicas: 0` for all indices. Without this, newly created indices are yellow (unassigned replicas) on a single-node cluster.

### `kibana`

- Installs `kibana`
- Fetches the `kibana_system` password from SSM and injects it into the Kibana keystore (`kibana-keystore add elasticsearch.password`) — the password does not appear in `kibana.yml`
- Renders `kibana.yml` with TLS config, Elasticsearch host, and Fleet pre-configuration
- Enables and starts `kibana` via systemd
- Waits for Kibana's `/api/status` endpoint to respond before the `fleet_server` role runs

### `fleet_server`

Runs on the Kibana VM after `kibana` is healthy. Responsibilities:

- Installs `elastic-agent`
- Checks `elastic-agent status` — all bootstrap steps below are skipped if the agent is already enrolled (idempotent guard)
- Fetches the `elastic` password from SSM
- Creates a Fleet Server service token via the Elasticsearch API (`POST /_security/service/elastic/fleet-server/credential/token/...`)
- Bootstraps Fleet Server (`elastic-agent install --non-interactive --fleet-server-*`) using the service token and the Kibana VM's TLS certificate
- Waits for Fleet Server to be healthy on port 8220
- Creates an APM Agent Policy in Kibana, adds the APM integration with TLS configuration, generates an enrollment token, and stores it in SSM at `/elastic/fleet/apm_enrollment_token`

Fleet Server and Kibana share the same TLS certificate (the Kibana cert has `fleet.<domain>` as a SAN).

### `apm_server`

- Installs `elastic-agent`
- Checks `elastic-agent status` — enrollment is skipped if already done (idempotent guard)
- Fetches the APM enrollment token from SSM
- Enrolls the agent with Fleet Server (`elastic-agent install --non-interactive --url=https://fleet.<domain>:8220 --enrollment-token=...`)
- Waits for the agent to report `HEALTHY`

Fleet then pushes the APM integration configuration (including port, TLS cert paths, and ES output) to this agent automatically.

## Playbook Ordering

`site.yml` imports the three playbooks in sequence. Each stage includes a health-check wait before the next begins:

```
elasticsearch.yml
  └─ common → elasticsearch
       └─ [waits for :9200 to respond]

kibana.yml
  └─ common → kibana → fleet_server
       └─ [waits for :5601] → [waits for :8220]

apm_server.yml
  └─ common → apm_server
       └─ [waits for agent HEALTHY]
```

This ordering is mandatory. Fleet Server requires Elasticsearch and Kibana to be up. The APM agent requires Fleet Server to be up and the enrollment token to exist in SSM.

## Idempotency

Re-running a playbook is safe:

- The EBS format step is skipped if `blkid` detects an existing filesystem.
- The `fleet_server` and `apm_server` roles check `elastic-agent status` before bootstrapping. If the agent is already enrolled, the bootstrap and all downstream SSM writes are skipped.
- Elasticsearch password resets and SSM writes run on every execution — re-running rotates the `elastic` and `kibana_system` passwords and updates SSM. If Kibana is already running with the old password, a re-run of `kibana.yml` is required afterwards to sync the keystore.

## Inventory Format

`ansible/inventory/hosts.yml` is generated by `terraform apply`. It uses EC2 instance IDs as host names and the `community.aws.aws_ssm` connection plugin:

```yaml
all:
  vars:
    aws_region: <region>
    domain_name: <domain>
  children:
    elasticsearch:
      hosts:
        i-xxxxxxxxxxxxxxxxx:
          ansible_connection: community.aws.aws_ssm
          ansible_aws_ssm_region: <region>
          private_ip: <ip>
          fqdn: elasticsearch.<domain>
          ssm_role_name: elasticsearch
    kibana:
      ...
    apm_server:
      ...
```

`ssm_role_name` drives the SSM Parameter Store paths in the `common` role (`/elastic/<ssm_role_name>/tls/*`).
